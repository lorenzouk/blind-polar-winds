🏟️ Atlas Arena: How It All Fits Together

The big picture

Atlas Arena is four cooperating pieces: a website where you sign in and find teammates, a game that runs in your browser, a game server in the cloud that runs the actual match, and Supabase — the shared backend that stores everything and runs the small backend functions.

The four parts

🖥️ The website — atlas-platform

This is atlasarena.com. You sign in, find two teammates, see when the next challenge starts, and check leaderboards and results. It never runs the game itself — its job is to get you into a match with the right credentials, then show you everything that came out of it.

🎮 The game — polar-winds-web

Polar Winds runs in your browser. You and two teammates are RED, GREEN and BLUE, painting a grid, collecting items and avoiding enemies — with built-in voice chat. Your browser draws the game and sends your key presses; it doesn't decide what happens.

⚙️ The game server — the authority

A program in the cloud that runs the one real copy of the match. Every key press from every player is sent to it; it decides what actually happens and sends the updated game state back to all three screens. One shared truth means nobody can cheat and everyone stays in sync.

🗄️ Supabase — the shared backend

One cloud service holding the database (accounts, teams, every session and score), logins, and file storage (replays, audio). It also runs dozens of small backend functions: create game tokens, match players, compute rankings, send emails.

One game, step by step

You sign in. The website checks your account with Supabase (email magic link or Discord).
You find teammates. You press Play and a backend function puts you in a queue until 3 players are ready. Voice chat starts while you wait in the lobby.
The website requests a game token. A backend function creates a signed token containing who you are and which match you're joining. It expires in 60 seconds, so it can't be stolen or reused.
Your browser moves to the game's address and presents the token. The game verifies it with Supabase and lets you in.
You play. For up to 30 minutes, every key press goes to the game server, which updates the one real copy of the game and sends the changes back to all three screens. Meanwhile a transcription service writes down what everyone says on voice chat.
Game over. The game server saves your score and a full replay to Supabase. The voice transcripts are saved too.
Back to the website. It shows your results and leaderboards — and later, a detailed report on how your team played and communicated.
💡 The big idea: the website and the game are two separate apps that never share a screen — your browser is handed from one to the other with a 60-second token, and both read and write the same Supabase backend. That's the whole architecture in one sentence.

The pieces, with real names

Three codebases do the work. atlas-platform is the website. polar-winds-web contains both the game's browser app (a thin “harness” shell plus the actual game visuals) and the authoritative game server. Everything durable lives in Supabase (database + auth + file storage + ~80 serverless “edge functions”), and LiveKit carries team voice.

🕹️ Harness vs. game code

Inside the game client, two layers are deliberately kept apart. The harness is everything around the gameplay: starting up, redeeming the token, talking to Supabase, matchmaking, voice, reconnecting, and returning to the platform. It doesn't know anything about how Polar Winds is played. The game code is only the gameplay: visuals, entities, input. Because the harness is game-agnostic, other games reuse the same pattern — new gameplay, same surrounding machinery.

🧰 The harness — polar-winds-web/client

Boot.tsx — the entry point. Reads ?token=… from the URL, calls the verify-game-token edge function to redeem it, learns who you are and which Supabase environment (dev/stage/prod) to talk to, and asks for mic permission.

Lobby.tsx + useGameLobby — matchmaking and the pre-game room: fetches the challenge seed (get-polarwinds-seed), joins the queue, gates the start on the challenge countdown.

Index.tsx — the session manager: connects to the Colyseus room, funnels server state into React, handles disconnects with exponential-backoff reconnection, and when it's all over sends you back to the platform via the returnUrl (with the result attached as query params).

Hooks — usePlatformVoice (LiveKit voice), useCanvasVideoPublish (publishes the game canvas for recording), session storage helpers.

🎨 The game code — client/src/screens

GameScreen.tsx — a Three.js / React Three Fiber canvas: top-down orthographic camera, bloom effects, the grid floor, and your keyboard/mouse input.

Entities — Player.tsx (glowing RED/GREEN/BLUE cubes), EnemyEntity.tsx, 7 collectible types (network, box, equilibrium, clone, vantage, galaxy, polyomino) spawned by a config-driven CollectibleFactory.

Crucially: no game rules live here. The client renders whatever state the server says is true and sends inputs up — which is exactly what makes the game cheat-proof and perfectly synced for all 3 players.

⚙️ The game server — polar-winds-web/server

A Node/Express + Colyseus app on Railway. GameRoom.ts owns the official GameState (players, grid colors, collectibles, enemies, scores, stage 1–8, 30-min clock, RNG seed), simulates moves and enemy AI, scores connected regions with BFS, and runs votes (clear board / abandon).

Colyseus auto-syncs state as tiny diff patches over WebSocket — clients get only what changed. Messages up: move, ping, dev commands. In solo mode, a bot fills the other two seats.

At game end it reports back to Supabase: end-polarwinds-session (final score + replay telemetry), set-polarwinds-highscore along the way, set-polarwinds-milestone for unlocks.

🖥️ The Platform — atlas-platform

React 18 + TypeScript + Vite + Tailwind, with React Router, Zustand and React Query. Auth is Supabase (magic-link email or Discord OAuth).

Pages: Dashboard, Play, Schedule, Challenge lobbies, Leaderboards, Profile, Performance (your skill report), and Admin tools. It also hosts the source of all the Supabase edge functions in supabase/functions/.

The launch moment lives in GameEmbed.tsx: call generate-game-token → receive a 60-second JWT + the game's URL → window.location.href = gameUrl + "/boot?token=…". A full hand-off of the browser tab — no iframe — and the token carries everything the game needs (identity, mode, challenge, competition, team code, admin/dev flags).

🎙️ Voice & transcription — LiveKit + transcriber/

LiveKit Cloud hosts voice rooms for the lobby and the game (token minted by the generate-livekit-token edge function).

The transcriber is a LiveKit Agents worker (Node) that auto-joins each game room, runs every player's audio through Deepgram speech-to-text, uploads {sessionId}/{color}.game.transcript.json to Supabase storage, then calls analyze-game-transcripts to generate team-communication feedback shown on the Performance page.

🗄️ Supabase — one stack, three environments

Three parallel projects — dev, stage, prod — each with: Auth (logins, JWTs), Postgres (users, teams, syndicates, sessions, challenge maps, matchmaking queue, skill scores…), Storage (replays, audio, transcripts, reports), and ~80 edge functions (Deno) that hold nearly all backend logic. The game token tells every component which environment to use, so dev games never touch prod data.

🏆 After the game: the score pipeline

A raw score isn't the end of the story. Once the game server files the session, a chain of edge functions turns it into meaning: process-session normalizes scores so different challenges are comparable; calculate-rankings updates every player's EloMMR rating for the leaderboards; and the Atlas Skill Engine (assess-session) replays the telemetry and reads the voice transcripts, runs them through Claude, and writes 37 skill scores per player across 6 categories — which generate-report can render into an emailable report. That's what the Performance page on the platform displays.

🧭 Who lives where

Repos → running things

atlas-platform → the website (static React app) plus the source of all Supabase edge functions (supabase/functions/).

polar-winds-web → one repo, three deployables: the game client (served from the game URL), the Colyseus server (Railway, same domain via WebSocket), and the transcriber worker (LiveKit Agents).

Three environments, one switch

dev / stage / prod are three separate Supabase projects. The game token says which one to use, so the harness, game server and transcriber all aim at the same environment for a given session — dev games can never scribble on prod data.

The same platform → token → harness pattern powers the other games on the platform too.

📖 Tiny glossary

Colyseus
A multiplayer game-server framework for Node.js. You define a “room” with typed state; it syncs that state to all connected players automatically, sending only what changed (diff patches).
Edge function
A small serverless function (here: Deno scripts hosted by Supabase) that runs one job on demand — “print a token”, “end a session”, “compute rankings”. The project has ~80 of them; they are the entire backend API.
JWT (the game token)
A signed token that can't be forged. The 60-second game token carries your identity and the game config (mode, challenge, competition, team code, dev/admin flags) across the website → game hand-off.
Supabase
Postgres database + auth + file storage + edge functions, as one hosted service. The single source of truth for everything durable.
LiveKit
Hosted real-time audio/video (WebRTC). Powers team voice chat, and its “Agents” framework runs the transcriber bot that joins each game as a silent listener.
EloMMR
A rating system (chess-Elo style) that updates each player's skill rating from normalized session scores; feeds the leaderboards.
Replay / telemetry
The event-by-event record of a session the game server uploads at the end — what the skill engine later “re-watches” to score 37 skills.
React Three Fiber
React bindings for Three.js (WebGL 3D). The game's visuals are React components that render into a 3D canvas.
Railway
The cloud host the game (client + Colyseus server) is deployed on; the platform reaches it via one URL (VITE_GAME_URL).
💡 The design in one sentence: a thin website and a thin game client, an authoritative game server in between, and one Supabase backend per environment that every piece reads and writes through ~80 small, single-purpose functions.
